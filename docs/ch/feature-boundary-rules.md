# 功能与源码文件划分规则

## 目标

在新增能力之前，先判断两件事：

1. 这个能力属于已有功能，还是一个新功能
2. 这个能力应该落到已有源码文件，还是新建源码文件

这一步必须先做，再开始实现与维护 `.coai`。

## 一、功能划分规则

功能边界按“场景事件的逻辑连续性”定义，而不是按代码文件定义。

判断顺序：

1. 先看这个能力服务的是哪个场景事件
2. 再看它是否只是这个场景事件中的一个新入口、新分支或新阶段
3. 只有当场景目标、触发时机、处理链路明显不同，才升级为新功能

### 属于已有功能的情况

- 只是已有场景事件新增了一个触发入口
- 只是已有流程新增了一个阶段
- 只是已有流程新增了一个分支或异常处理
- 新能力的目标仍然服务于原功能主目标

### 需要拆成新功能的情况

- 场景目标已经变化
- 触发时机已经独立
- 处理链路已经不再依赖原功能主流程
- 即使移除原功能，新能力仍然能单独成立

## 二、源码文件划分规则

代码文件边界按“实现职责隔离”定义，而不是按功能文档定义。

判断顺序：

1. 先确定功能归属
2. 再判断实现是否应新建源码文件

### 应新建源码文件的情况

- 新能力会显著增加已有文件复杂度
- 新能力有独立的输入、输出或编排逻辑
- 新能力需要避免频繁修改已稳定的实现
- 新能力虽然属于旧功能，但实现职责已经可以单独维护

### 可以继续放在原文件的情况

- 只是很小的参数扩展
- 只是局部逻辑补丁
- 不会明显增加文件职责混乱

## 三、`.coai` 维护规则

新增能力前，必须先完成下面判断：

1. 功能归属
2. 源码文件归属
3. 是否需要新增或调整 `@coai anchor`
4. 是否需要补充该功能文档的“补充说明”
5. 是否需要更新对应 mapper / node

如果功能归属仍是旧功能：

- 继续维护原功能文档
- 不新建新的主功能文档
- 但允许新建源码文件

如果功能归属已经变成新功能：

- 新建功能文档
- 新建对应 mapper / node
- 再决定源码文件划分

## 四、共享代码与多语义 anchor 规则

当多个业务节点命中同一段共享代码时，先不要立刻把它升格成 `utils/common` 功能。

默认规则：

- 允许在同一代码段附近维护多条不同的 `@coai anchor`
- 每条 anchor 都对应一个不同的业务语义节点
- 只要这些节点对开发者来说仍然是不同的功能入口，就不应该被粗暴合并

什么时候才升格为 `utils/common` 功能：

- 当共享能力本身已经成为开发者需要单独理解的认知入口
- 当开发者后续经常直接从“公共校验”“公共缓存”“公共错误处理”这类视角理解项目
- 当即使脱离原业务功能，这个共享能力本身也值得有独立功能文档

什么时候不要升格：

- 只是因为代码位于 `utils`、`common`、`shared` 目录
- 只是因为多个功能复用了同一个工具函数
- 只是因为多个 mapper 恰好落到同一段代码

触发时机：

1. agent 准备新增或修改 mapper 条目时
2. agent 准备在共享代码中补 `@coai anchor` 时
3. agent 准备把某段共享实现抽象为 `utils/common` 功能文档时

默认决策：

- 如果共享代码只是实现复用，保留原业务节点，并允许多条不同 anchor 靠近同一段代码
- 如果共享代码本身已经成为认知对象，再新增公共功能文档
- 新增公共功能文档不替代原业务节点，而是新增一层共享能力认知

## 五、当前仓库中的判断示例

### `navigation`

场景事件：

- 双链悬停查看与跳转

实现文件可以拆成：

- `navigation.ts`

但功能仍然只有一个：

- `navigation`

### `git-sync`

场景事件：

- 维护 `.coai/mapper`
- 基于 Git 增量进行校正与冻结判断

如果新增“提交前先跑一次检查”，它仍然属于 `git-sync`，因为：

- 目标仍是做 CoAI 维护与检查
- 只是新增了一个触发入口
- 不是一个独立的新业务场景

但源码上建议新建：

- `preCommitCheck.ts`

原因：

- 避免把触发入口与扫描核心都堆进 `gitSync.ts`
- 降低对已稳定同步逻辑的频繁修改

### `bug-repair`

场景事件：

- bug 检测
- bug 记录
- Problems 面板显示
- 修复流转
- 归档与 reopen

虽然实现跨：

- `gitSync.ts`
- `bugLog.ts`
- `repair.ts`

但主功能仍然只有一个：

- `bug-repair`

### `cli`

场景事件：

- 提供终端命令入口
- 为 Git hook 提供可调用执行面
- 承接未来更多非 VS Code 宿主下的 CoAI 子命令

因此 `cli` 应单独作为功能存在，而不是只把它当成某个已有功能的附属实现细节。

当 `cli` 已经进一步承担：

- 终端命令入口
- Git hook 生命周期管理
- 仓库初始化入口
- 团队级模板资产承接

它仍然属于同一个 `cli` 主功能，因为这些能力都服务于同一条“非 VS Code 宿主下的 CoAI 执行与分发场景”。

## 六、提交前检查的接入原则

“commit 前先跑一次 CoAI 检查”当前归属到 `git-sync`，不是一个新主功能。

推荐实现顺序：

1. 新建 `preCommitCheck.ts`
2. 提供显式命令入口
3. 在 `git-sync` 功能文档中补“提交前检查入口”
4. 后续如果真的发展成提交放行策略、hook、staged 控制，再评估是否升级为新功能

## 七、关于 VS Code 与 CLI 的现实边界

开发者可能通过两种方式提交：

- 终端里执行 `git commit`
- VS Code Source Control 面板提交

当前更可靠的结论是：

- 只靠 VS Code 扩展，无法稳定拦截所有 CLI `git commit`
- 如果未来要统一覆盖 CLI 与 VS Code 两种提交方式，应优先走 Git hook

这意味着：

- “显式命令入口”适合当前阶段
- “真正的提交前统一检查”适合后续通过 `pre-commit` hook 落地
- 为了让 hook 可调用，需要单独建设 `cli` 功能，而不是只依赖 VS Code 命令

如果以后需要支持 hook，最好把检查核心做成可复用的独立逻辑，而不是只绑在 VS Code 扩展运行时里。
