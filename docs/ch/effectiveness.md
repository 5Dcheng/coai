# CoAI 有效性测试结果

## 范围

本文整理 TeamDesk Lite 实验中目前能支撑 CoAI 产品有效性的证据。

这是一份产品层面的初步佐证，不是最终学术 benchmark。当前数据足以支持的核心判断是：

> CoAI 能帮助 AI coding agent 沉淀并复用项目认知，减少维护任务中的重复源码探索，并把功能理解连接到真实代码入口。

但不应表述为：

> CoAI 在所有任务中都一定更省 token、更少读文件，或可以替代源码验证。

## 测试对象

TeamDesk Lite 是一个轻量工单系统，技术栈为：

- Vite + React + TypeScript 前端
- Express + TypeScript 后端
- 本地 JSON 文件存储
- 功能包括工单列表、新建、详情、元数据编辑、状态流转、评论、Activity timeline 与 Dashboard 统计

对照组：

| 条件 | 说明 |
|---|---|
| baseline | 新窗口直接读取源码、README、测试与配置来理解项目。 |
| AGENTS.md | 长上下文窗口生成单文件 `AGENTS.md`，新窗口优先读取它，再按需验证源码。 |
| CoAI | 新窗口优先读取 `.coai` feature cognition 与 mapper，再按需验证源码。 |

## 关键结论

最可信的结论是：

> CoAI 不是最低成本的单次项目总结工具；CoAI 的价值在于提供按功能组织、可定位源码入口、可随维护回流更新的项目认知层。

换句话说：

- baseline 可以完成任务，但每个新窗口都要重新建立项目认知。
- AGENTS.md 的新窗口使用成本很低，但它是手动维护的单文件上下文资产，生成和更新成本可能很高。
- CoAI 有初始化和维护成本，但提供 feature cognition、mapper/anchor 与认知回流，更适合长期维护和局部功能变更。

## R31 新窗口项目认知测试

R31 测试新窗口理解已完成项目的成本。

GLM-5-turbo 账单数据：

| 条件 | billed tokens | 费用 CNY | 读取文件数 | 源码读取数 |
|---|---:|---:|---:|---:|
| baseline R31 | 299,655 | 0.7905314 | 26 | 26 |
| CoAI R31 | 174,740 | 0.508423 | 31 | 8 |
| AGENTS.md R31 | 61,919 | 0.151387 | 4 | 3 |

解释：

- baseline 通过广泛读取源码来重建项目认知。
- CoAI 把源码读取从 26 降到 8，本次账单 billed usage 也下降。
- AGENTS.md 在已有文件且内容足够新鲜的情况下，新窗口读取成本最低。

但 AGENTS.md R31 只统计“使用阶段”，不是完整方案成本。AGENTS.md 至少有三类成本：

```text
AGENTS.md 生命周期成本 =
  A00 首次生成成本
  + 每次新窗口读取成本
  + 项目推进后的更新成本
```

其中 A00 首次生成成本已经测到，后续增量更新成本尚未测。

AGENTS.md A00 首次生成成本：

| 成本项 | billed tokens | 费用 CNY |
|---|---:|---:|
| AGENTS.md A00 生成 | 707,449 | 1.8105794 |
| AGENTS.md R31 使用 | 61,919 | 0.151387 |
| 首次使用合计 | 769,368 | 1.9619664 |

所以 AGENTS.md 的正确解释是：

1. 如果已经有一份新鲜、高质量 AGENTS.md，新窗口使用非常便宜。
2. 如果只使用一次，生成成本很高，总成本不低。
3. 如果项目持续推进，每次开新 chat 前还需要更新 AGENTS.md；更新可能依赖旧长上下文，也可能重新扫描源码。前者可能有过期信息和上下文污染，后者会重新带来扫描成本。
4. CoAI 也有维护成本，但它更聚焦在当前 feature 的认知回流，而不是每次把整个项目压缩成单文件交接摘要。

## 维护任务中的证据

CoAI 更强的信号出现在功能级维护任务，而不是一次性项目总结。

baseline2 与 post-hoc CoAI 在 R26-R30 的直接对照：

| 指标 | baseline2 R26-R30 | CoAI R26-R30 |
|---|---:|---:|
| 总读取文件数 | 47 | 56 |
| 源码/配置/测试/数据读取数 | 46 | 21 |
| `.coai` 文件读取数 | 0 | 35 |

解释：

- CoAI 没有减少总读取文件数。
- CoAI 显著减少源码读取，把一部分源码探索转移为读取高密度项目认知资产。

典型任务：

| 任务 | CoAI 表现 |
|---|---|
| 定位评论 activity 文案 | 先读 comment/activity cognition 与 mapper，再验证一个源码文件。 |
| 修改评论摘要长度 | 修改源码后同步更新对应 feature cognition。 |
| 分析 `closed -> reopened` 状态规则 | 从 status-transition 认知出发，覆盖前端、后端、统计、测试与数据语义。 |

## 为什么能支撑 CoAI 有效

当前证据支持：

1. CoAI 能减少新窗口和维护任务中的重复源码探索。
2. CoAI 按功能组织项目认知，而不是只按目录结构。
3. mapper/anchor 能把功能语义连接到真实源码入口。
4. CoAI 支持认知回流：代码变更后，稳定项目理解也能同步更新。
5. CoAI 更适合长期维护、多轮接手、局部功能定位和影响分析。

## 当前不能过度声明

当前证据不支持：

- CoAI 总是降低 token。
- CoAI 总是降低总文件读取。
- CoAI 初始化成本可以忽略。
- CoAI 可以替代源码验证。
- CoAI 一定比新鲜 AGENTS.md 的单次项目总结更便宜。
- CoAI 对同一 chat 内的小改动一定更轻。

## AGENTS.md 与 CoAI 的关系

AGENTS.md 是非常强的对照组，但不是 CoAI 的完全替代。

| 维度 | AGENTS.md | CoAI |
|---|---|---|
| 形态 | 单文件摘要 | 结构化认知资产 |
| 维护方式 | 手动触发，通常在开新 chat 前更新 | 作为 feature 变更循环的一部分 |
| 代码定位 | 描述性提示 | mapper/anchor 定位 |
| 功能边界 | 通常是总结 | 显式 feature cognition |
| 更新风险 | 容易过期或受旧 chat 上下文污染 | 仍需维护，但有明确认知回流流程 |

AGENTS.md 让新窗口快速知道项目是什么。

CoAI 让 agent 围绕某个功能知道该读哪里、改哪里、更新哪份认知。

## 产品定位

CoAI 应定位为：

> 面向 AI coding agent 的项目认知层。

更具体地说：

> CoAI 把项目理解转化为仓库内的、按功能组织的、可定位源码入口的认知资产，让项目认知跨 chat、跨维护周期持续存在。

适用场景：

- 项目由多轮 AI coding 推进
- 新窗口需要快速接手，而不想重新扫源码
- 开发者希望功能语义与真实源码入口连接
- agent 需要局部修改功能，同时保持 feature 边界
- 代码变更后，希望稳定认知能同步更新，而不是只留在聊天记录里

## 当前限制

这组实验仍是初步证据：

- 主要来自一个测试项目
- 项目规模中小
- GLM token 数据样本数有限
- 是 post-hoc CoAI 接入，不是从 R01 原生 CoAI 开发
- AGENTS.md 后续增量更新成本还未测
- 尚无真实开发者用户研究数据

公开表达应保留这些边界。

## 后续建议

如果后续还有时间，优先测试：

1. 功能级任务三方对照：
   - 修改评论 activity 文案
   - 修改状态流转规则
   - 修改 Dashboard 统计语义
   - 新增 Ticket 字段并同步前后端测试
2. 测 AGENTS.md 在多轮开发后的增量更新成本。
3. R31 token 测试重复 2-3 次，降低单次偶然性。
4. 在更大项目中测试，因为大项目的源码探索成本会更明显。
