# CoAI 与 Spec 类系统的区别

## 目的

这份文档用于说明 CoAI 与 Spec-Driven Development、`spec-kit`、以及类似 spec / plan 工作流之间的关系与边界。

目的不是否认两者有交叉，而是明确各自的主职责，避免 CoAI 逐渐漂成一个错误的产品类别。

## 最短结论

两者有交叉，但重心不同。

- spec 类系统更关注规格、执行计划、任务拆解、依赖顺序与工程控制
- CoAI 更关注功能语义、状态驱动的项目认知、语义到代码的连接，以及实现后的认知回流

一句话概括：

- spec 类系统是过程驱动的执行编排框架
- CoAI 是状态驱动的项目认知层

## 交叉点

两者都可能出现在 coding 开始之前。

两者都可能帮助回答这些问题：

- 这个功能是什么
- 理想路径是什么
- 有哪些约束
- 下一步要做什么

所以 CoAI 不是“完全不管前置设计”，spec 系统也不是“完全不管认知”。

真正的区别在于：它们各自的输出，长期想保留下来的到底是什么。

## 核心区别

### spec 类系统

主要关注：

- 如何组织并控制一次变更

典型输出：

- spec
- plan
- tasks
- design
- constitution / rules
- checkpoint / test gate

主要价值：

- 在实现前降低模糊性
- 组织执行顺序
- 让多步骤工程工作更可控

### CoAI

主要关注：

- 项目在持续演进时，如何仍然保持可理解

典型输出：

- `project/<module>/<feature>.md`
- `node/*.node.json`
- `mapper/*.mapper.json`
- `current.md`
- bug log
- `@coai anchor`

主要价值：

- 保留功能含义
- 稳定功能认知入口
- 把语义挂回真实代码
- 让实现完成后的认知重新回流到项目中
- 避免长期 AI coding 项目重新变成黑盒

## 过程 vs 状态

这是最关键的区别。

spec 类系统通常更偏过程导向：

- 定义变更
- 拆解任务
- 安排顺序
- 执行
- 验证

CoAI 更偏状态导向：

- 定义功能语义
- 保留当前仍有效的认知
- 把功能含义绑定到实现
- 随实现变化更新项目状态

CoAI 的前提是：项目是以状态推进的，而不是靠保存通向当前状态的全部过程细节来推进的。

这也是为什么 CoAI 不应该变成完整的探索历史档案库。

## 为什么 CoAI 不该变成 `spec-kit`

如果 CoAI 继续扩展到完整的 plan/task 编排，它就会进入一个已经有更成熟工具的类别。

这会削弱 CoAI 最尖锐的差异化。

CoAI 最强的位置不是：

- “更轻量的 spec 系统”
- “可视化计划系统”
- “另一套任务拆解工作流”

CoAI 最强的位置是：

- 编码前的功能语义设计
- 编码中的语义到代码桥接
- 编码后的认知回流
- 长期项目的可理解性维护

## CoAI 在编码前能做什么

CoAI 在实现前也有明确价值。

它适合帮助定义：

- 理想路径
- 核心功能语义
- 流程中的关键节点
- 高层技术方向
- 未来要和代码绑定的语义锚点

这比完整执行 spec 更轻。

它更接近“语义级功能设计”，而不是“完整工程编排”。

## CoAI 在编码后做什么

这正是 CoAI 最独特的地方。

当代码实现并验证后，CoAI 可以：

- 通过 anchor 和 mapper 把功能含义挂回代码
- 通过 node 提供 Hover 认知
- 通过 git 增量维护 mapper 行号
- 通过 bug log 让认知层异常显式可见
- 通过 `current.md` 保留当前仍有意义的项目状态

这些不是 spec 类系统的核心职责。

## 产出物区别

### spec 类产出物

优化目标：

- 执行清晰度
- 顺序控制
- 依赖管理
- 工程严谨性

它主要回答：

- 我们该做什么
- 按什么顺序做
- 带着什么约束做
- 怎么验证这次变更

### CoAI 产出物

优化目标：

- 功能语义清晰
- 项目理解效率
- 代码可到达性
- 长期认知连续性

它主要回答：

- 这个功能到底是什么意思
- 它具体落在哪些代码里
- 哪些实现点最值得理解
- 当代码演进后，哪些认知必须继续留下来

## 两者如何串联

两者可以互补。

一种合理链路是：

1. 先用 CoAI 表达功能语义与理想路径
2. 如有需要，再用 spec / plan 系统把它转成执行结构
3. 进入 coding
4. 完成验证
5. 再由 CoAI 接收认知回流，并把稳定语义挂回代码

可以压成一句：

`CoAI -> spec/plan -> coding -> validation -> CoAI`

但这不意味着 CoAI 应该吸收完整的 spec / planning 层。

## 非技术用户入口

CoAI 对非工程师或非技术用户有一个很强的入口价值。

很多人知道：

- 自己想要的工作流程
- 最终想得到的结果
- 事务的大致步骤
- 常见异常与风险

但他们并不知道技术语言。

CoAI 允许这类工作流认知先被表达为可被 agent 使用的功能语义结构，然后再进入技术实现。

这和很多 spec 系统不同。后者往往默认：需求已经准备好被工程化地正式表述。

## 长期是否需要

spec-heavy 的工作流在这些场景里非常有价值：

- 大团队
- 高风险系统
- 强治理环境
- 依赖复杂的大型工程

但它并不是所有 AI-native 项目的长期必需品，尤其是在这些场景下：

- 单人或小团队
- 产品仍在快速生长
- 强 coding agent 已经具备较强内部规划能力
- 真正的问题开始变成“理解漂移”，而不是“任务不会拆”

在这些情况下，即便 formal spec 流程减少或可选，CoAI 仍然可以长期有价值。

## 最终定位

CoAI 不应该把自己定位为 `spec-kit` 的直接竞品。

更合适的定位是：

- spec 类系统负责组织执行
- CoAI 负责保留项目认知

或者更锋利一点：

- spec 类系统帮助决定如何执行一次变更
- CoAI 帮助持续保持系统可理解

## 推荐的一句话定义

当需要明确区分时，可以使用：

**CoAI 是 AI 原生开发中的状态驱动项目认知层，不是过程驱动的执行编排系统。**
