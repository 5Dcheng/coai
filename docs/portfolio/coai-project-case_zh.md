# CoAI 项目案例：面向 AI Coding 的项目认知层

## 1. 项目定位

CoAI 是面向 AI Coding agent 的项目认知层。

它解决的不是“让 AI 写代码”这个问题，而是 AI 已经能快速写代码之后，项目如何继续被人和 agent 理解、定位和维护的问题。

一句话：

> CoAI 把项目的功能目标、输入输出、关键流程、约束边界和源码入口沉淀为可维护的项目认知地图，让 AI 生成的项目不再快速黑盒化。

## 2. 为什么这是一个真实问题

在多轮 AI Coding 中，我观察到一个稳定问题：

代码生成速度提升后，项目理解反而快速下降。

典型表现：

- agent 每次换新 chat 都要重新读 README、源码、测试和配置。
- 开发者知道“功能大概能跑”，但难以从功能视角理解项目结构。
- 功能逻辑散落在多个源码文件中，运行时结构对程序是合理的，但对人和 agent 不一定好索引。
- chat 历史成为隐形上下文资产，一旦窗口切换，上下文丢失或漂移。
- 多步骤维护任务中，agent 容易先全局扫源码，再逐步猜测修改入口。

我将这个问题抽象为：

> AI Coding 项目缺少一层稳定、可定位、可回流的“项目认知地图”。

这也是 CoAI 的核心产品机会。

## 3. CoAI 的新增能力：项目认知地图

CoAI 的核心价值首先是能力，而不仅是效能。

它新增了一种仓库内的项目认知资产，让开发者和 agent 可以从功能语义进入项目，而不是只能从目录树或聊天记录进入项目。

### 3.1 面向开发者的能力

CoAI 帮助开发者看到：

- 当前项目有哪些功能模块。
- 每个功能的目标是什么。
- 输入和输出是什么。
- 关键节点流程是什么。
- 哪些约束不能破坏。
- 出问题时应该从哪个源码入口开始看。

这缓解的是“项目快速黑盒化”问题。

### 3.2 面向 agent 的能力

CoAI 帮助 agent 在写代码前获得高语义上下文：

- feature goal：本次功能到底要解决什么。
- boundary：哪些属于当前 feature，哪些不该碰。
- constraints：哪些业务约束不能破坏。
- source entry：该从哪些源码入口验证实现。
- cognition backflow：代码改完后，哪些稳定认知需要更新。

这不是普通 README 能稳定提供的内容。

## 4. 核心机制

CoAI 将项目认知拆成几类资产：

| 机制 | 作用 |
|---|---|
| `feature.md` | 记录功能目标、输入输出、关键流程、约束边界、异常路径和维护风险。 |
| `mapper.json` | 将功能语义节点映射到源码中的 anchor/file/line。 |
| `@coai anchor` | 在关键源码入口处放置稳定语义锚点。 |
| `current.md` | 记录动态项目状态和近期进展，不替代 feature 文档。 |
| cognition backflow | 代码变更后，同步更新稳定项目认知。 |

产品闭环：

```text
功能认知对齐
  -> mapper/anchor 确认源码入口
  -> agent 定向修改源码
  -> 验证
  -> 稳定认知回流 feature.md
```

CoAI 的重点不是取代源码。源码仍然是最终事实来源。

CoAI 的重点是：

> 让 agent 先从功能认知进入，再定向验证源码，而不是每次从全局源码探索开始。

## 5. 为什么 AGENTS.md 不等价于 CoAI

AGENTS.md 是一个很强的 baseline，也确实能降低新窗口单次理解成本。

但它本质上是 baseline 上额外增加的一次“人工/agent 总结步骤”：

```text
baseline 长上下文
  -> 手动要求 agent 生成 AGENTS.md
  -> 新 chat 读取 AGENTS.md
```

这意味着 AGENTS.md 不是免费、自动、持续更新的上下文资产。

### 5.1 AGENTS.md 的真实成本

在 TeamDesk Lite 测试中：

| 成本项 | billed tokens | 费用 CNY |
|---|---:|---:|
| AGENTS.md 首次生成 | 707,449 | 1.8105794 |
| AGENTS.md 新窗口使用 | 61,919 | 0.151387 |
| 首次使用合计 | 769,368 | 1.9619664 |

AGENTS.md 新窗口使用确实很便宜，但生成本身成本很高。

更重要的是，项目继续推进后，它还需要更新：

```text
项目推进一段时间
  -> 准备开新 chat
  -> 旧 chat 显式更新 AGENTS.md
  -> 新 chat 再读取
```

如果依赖旧长上下文更新，可能混入过期信息和无关内容；如果重新扫描源码更新，成本又会上升。

### 5.2 CoAI 与 AGENTS.md 的本质差异

| 维度 | AGENTS.md | CoAI |
|---|---|---|
| 形态 | 单文件上下文总结 | 按 feature 组织的认知资产 |
| 维护方式 | 手动要求 agent 更新 | 作为功能变更循环的一部分 |
| 信息结构 | 容易混合架构、风险、历史、入口 | 目标、边界、流程、约束、入口分层组织 |
| 源码定位 | 描述“大概在哪” | mapper/anchor 指向关键源码入口 |
| 长期维护 | 容易过期或膨胀 | 可按 feature 回流更新 |

因此，AGENTS.md 是 CoAI 必须面对的强对照组，但不是 CoAI 的替代品。

更准确的定位是：

> AGENTS.md 让新窗口快速知道项目是什么；CoAI 让 agent 围绕某个功能知道该读哪里、改哪里、更新哪份认知。

## 6. 验证设计

我设计了 TeamDesk Lite 作为真实 AI Coding 验证项目。

项目包含：

- 工单列表
- 新建工单
- 工单详情
- 元数据编辑
- 状态流转
- 评论
- Activity timeline
- Dashboard 统计
- JSON 文件存储

对照组：

| 条件 | 含义 |
|---|---|
| baseline | 新窗口直接读源码、README、测试和配置。 |
| AGENTS.md | baseline 上额外生成单文件上下文，新窗口优先读取。 |
| CoAI | 新窗口优先读取 `.coai` feature cognition 和 mapper，再按需验证源码。 |

测试任务：

1. 新窗口项目认知恢复。
2. 评论 activity 文案定位。
3. 小维护改动。
4. 状态流转规则影响分析。
5. 项目维护总结。

## 7. 能力验证：CoAI 提供项目认知地图

在测试中，CoAI 形成了一个按功能组织的项目认知地图：

```text
ticket-list
ticket-create
ticket-detail
ticket-meta-edit
status-transition
comment
activity-timeline
dashboard
json-storage
api-routing
```

这对 agent 和开发者都有直接价值：

- 不再只通过源码目录理解项目。
- 可以从功能目标进入维护任务。
- 能识别 feature 边界，减少误改。
- 能定位关键源码入口。
- 能在代码变化后回流稳定认知。

这是 CoAI 的核心能力增量。

即使不只看 token，单这一项能力也有产品价值：

> 它让 AI coding 项目从“代码能跑但难理解”，变成“功能认知、代码入口和维护约束可被持续读取”。

## 8. 效能验证：减少重复源码探索

### 8.1 新窗口项目认知任务

R31 新窗口项目认知测试：

| 条件 | billed tokens | 费用 CNY | 源码读取数 |
|---|---:|---:|---:|
| baseline | 299,655 | 0.7905314 | 26 |
| CoAI | 174,740 | 0.508423 | 8 |
| AGENTS.md 使用阶段 | 61,919 | 0.151387 | 3 |

注意：这里的 `AGENTS.md 使用阶段` 只表示“文件已经存在且足够新鲜时，新窗口读取它”的成本，不代表 AGENTS.md 方案的完整成本。

AGENTS.md 方案至少包含三部分：

```text
AGENTS.md 生命周期成本 =
  A00 首次生成成本
  + 每次新窗口读取成本
  + 项目推进后的更新成本
```

在本次测试中，AGENTS.md A00 首次生成本身消耗 **707,449 billed tokens / 1.8105794 元**。因此不能只用 `61,919 tokens / 0.151387 元` 判断 AGENTS.md 方案整体优于 CoAI；它说明的是“已有一份新鲜 AGENTS.md 时，新窗口接手很便宜”。

AGENTS.md 还需要人工触发维护：项目推进一段时间后，如果准备开新 chat，旧窗口通常需要显式更新 AGENTS.md。这个更新过程可能依赖长上下文总结，也可能重新读取大量源码；前者有过期信息和上下文污染风险，后者会重新带来扫描成本。CoAI 的维护成本也存在，但它更聚焦在当前 feature 的认知回流，不需要每次把整个项目压缩成单文件交接摘要。

CoAI 相比 baseline：

- 源码读取数从 26 降到 8，下降约 **69.2%**。
- GLM billed usage 从 299,655 降到 174,740，下降约 **41.7%**。
- 费用从 0.7905314 元降到 0.508423 元，下降约 **35.7%**。

解释：

> 源码读取下降不是说“源码不重要”，而是说明 agent 从全局源码探索转向基于项目认知地图的定向源码验证。

读源码的好处仍然存在：源码是事实来源。

CoAI 的价值是让读源码更有目标。

### 8.2 维护任务对照

在 R26-R30 维护任务中：

| 指标 | baseline2 | CoAI |
|---|---:|---:|
| 总读取文件数 | 47 | 56 |
| 源码/配置/测试/数据读取数 | 46 | 21 |
| `.coai` 文件读取数 | 0 | 35 |

这组结果很重要：

- CoAI 不一定降低总读取文件数。
- CoAI 把大量源码读取替换为 `.coai` 高语义认知读取。
- 源码相关读取从 46 降到 21，下降约 **54.3%**。

因此，更准确的效能表达不是“少读文件”，而是：

> CoAI 将 agent 的探索从低语义源码扫描，转为高语义功能认知读取 + 少量源码验证。

## 9. CoAI 的产品结果

当前证据支持以下产品判断：

### 9.1 能力层面

CoAI 建立了 AI Coding 项目认知地图：

- 明确功能目标。
- 明确输入输出。
- 明确关键节点流程。
- 明确约束边界。
- 明确源码入口。
- 支持代码变更后的认知回流。

这直接解决项目快速黑盒化问题。

### 9.2 效能层面

在 TeamDesk Lite 测试中，CoAI：

- 降低新窗口项目认知中的源码探索。
- 降低本次 GLM billed usage。
- 在维护任务中减少源码/配置/测试/数据读取。
- 使局部功能定位更接近 feature 维度，而不是从文件树重新探索。

### 9.3 产品化层面

CoAI 的产品定位从“更省 token”收口为：

> 面向 AI Coding agent 的项目认知层，帮助长期项目保持可理解、可定位、可维护。

这个定位更稳，也更有差异化。

## 10. 我的贡献

### 问题抽象

- 从 AI Coding 多轮任务中识别项目黑盒化、上下文丢失、功能边界漂移等问题。
- 将问题抽象为“缺少项目认知层”，而不是简单归因于 agent 能力不足。

### 产品方案设计

- 设计 feature cognition、mapper、anchor、current.md 等机制。
- 明确 CoAI 与 README、Spec、AGENTS.md、IDE 索引的边界。
- 设计功能认知、源码定位和认知回流闭环。

### 评估设计

- 设计 baseline / AGENTS.md / CoAI 三组对照。
- 设计项目认知、局部维护、影响分析等任务。
- 使用源码读取数、GLM billed usage、任务质量、认知回流等指标评估效果。

### 产品定位修正

- 识别 AGENTS.md 是强 baseline，但不是 CoAI 的替代品。
- 将 CoAI 价值从单纯“更省 token”修正为“功能认知地图 + 源码入口 + 认知回流”。
- 保留证据边界，避免过度声明。
