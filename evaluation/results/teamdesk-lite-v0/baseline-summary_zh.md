# TeamDesk Lite Baseline 汇总

## 范围

- 条件组：baseline
- Agent：Trae SOLO Coder
- 模型：Kimi K2.6
- IDE 上下文能力：Trae 内置 workspace index 已启用
- 项目：TeamDesk Lite
- 主开发轮次：R01-R20，外加 1 个修复轮
- 新窗口维护轮次：R21-R25
- 人工验收：核心功能基本实现，手动测试通过

这不是裸 LLM baseline。它包含 Trae 的工作区索引能力和 agent 工具能力。

## 核心结论

baseline 成功完成了 TeamDesk Lite 项目。

因此，CoAI 评估不应该把问题表述为“AI coding 能不能完成这种项目”。在这个案例中，`Trae + Kimi K2.6 + workspace index` 已经可以完成。

CoAI 更有意义的评估问题是：它是否改善了开发与维护过程。

重点包括：

- 降低上下文重建成本
- 减少大范围探索式读文件
- 减少修复轮次
- 减少范围漂移
- 改善按功能导航项目的能力
- 改善新窗口上下文延续
- 降低项目完成后的开发者/agent 黑盒感

## R01-R20 开发结果

项目达到了实际可用状态：

- 工单创建、列表、详情、筛选可用
- 状态流转规则已实现
- 状态历史和 activity timeline 已实现
- 评论功能已实现
- 元数据编辑已实现
- Dashboard 统计已实现
- build 和 test 通过
- 人工测试确认核心功能基本可用

最终产品是可用的，但开发路径中存在早期不稳定和范围控制信号。

关键模式是：新窗口恢复项目认知的第一轮成本较高；一旦同一窗口建立了上下文，后续维护定位成本明显下降。

## 开发过程模式

### 早期阶段：R01-R05

agent 推进很快，也完成了大量工作，但同时提前实现了不少后续功能。

重要观察：

- R02 提前实现了许多后续后端功能。
- R03 同时实现了列表、新建、详情和 Dashboard 占位。
- R05 扩展到了元数据编辑、状态流转、历史、评论和 activity timeline。
- build/test/runtime 问题主要是在人工验收时才暴露。

R01-R05 人工检查发现了阻断问题：

- TypeScript build 失败。
- test 命令因为没有测试文件而失败。
- comments route 缺少 `addActivity`，导致后端运行时崩溃。
- Express route 参数类型存在错误。

### 修复阶段：R05-FIX

修复轮解决了阻断问题，并恢复了 build/test/dev 可用性。

这个轮次是重要成本信号：baseline 可以恢复，但在连续功能开发后需要专门返工修复。

### 稳定阶段：R06-R20

修复轮之后，agent 的表现明显更稳定。

较好的样本包括：

- R06 只补充状态流转测试，没有改产品行为。
- R09 聚焦实现筛选功能。
- R16 补充了有意义的基础测试覆盖。
- R17 用小范围改动修复筛选大小写问题。
- R19 做纯回归验证，没有改代码。
- R20 做项目维护盘点，没有改代码。

仍然存在的范围控制信号：

- R12 提前覆盖了 R13/R14 的 Dashboard 指标。
- R14 识别平均解决时间已经完成，但改动转向了全局导航。
- R15 识别 reopened 统计已经正确，但清空了 data JSON 文件。

## 新窗口维护结果

R21-R25 在新窗口中运行，用于测试项目认知恢复和维护上下文。

### R21：初始恢复

agent 输出了高质量的项目理解总结，但读取了 21 个文件。

这是 baseline 对 CoAI 最重要的信号：

> 没有显式功能认知层时，新窗口 agent 可以恢复理解，但方式是大范围读取源码、配置和测试文件。

### R22-R23：功能链路定位

R21 之后，agent 可以较高效地定位跨文件功能链路：

- R22 评论链路：读取 6 个文件
- R23 状态流转影响分析：读取 7 个文件

两轮回答都准确，并且是按功能组织的。

### R24：小维护改动

agent 完成了一个精准的小维护修改：

- 读取文件数：2
- 修改文件数：1
- test 通过
- build 通过

该修改只影响新生成的 `comment_added` activity message。

### R25：维护总结

agent 复用了 R21-R24 积累的上下文，只额外复查了 1 个文件。

这说明普通 chat 一旦支付了初始认知重建成本，也能在同一窗口内保持上下文。CoAI 对比时应重点看：CoAI 能否降低新窗口第一轮的初始重建成本。

## 开发者黑盒感观察

用户人工测试最终项目后确认核心功能基本可用。

同时，用户报告了维护层面的黑盒感：

- 项目功能可用，但功能认知并不直接可见。
- 行为分散在面向运行时的源码文件中。
- 即便知道可以从入口文件开始读，也不愿意手动重新拼接功能地图。
- 这会影响开发者维护项目时的耐心、信心和掌控感。

这是 CoAI 评估的核心。

源码组织对程序运行是正确的，但它不足以作为人和 agent 的功能导航层。

## Baseline 已经做得好的地方

baseline 很强。

- 它可以完成一个中小型产品。
- 它可以从 build/runtime 错误中恢复。
- 它在建立上下文后可以做精准维护。
- 它可以输出准确的功能链路解释。
- 它可以运行 build 和 test 验证。

这提高了 CoAI 的证明标准：CoAI 不能只是证明 agent 能完成任务。

## CoAI 的预期优势

CoAI 应围绕以下具体提升点评估。

### 1. 新窗口上下文恢复

预期提升：

- R21 这类项目理解任务读取更少源码文件
- 更快恢复功能地图
- 不必重新检查每个 route、page、test

主要对比：

- baseline R21 读取了 21 个文件
- CoAI R21 理想情况下应先读 `.coai` 认知资产，再选择性验证源码

### 2. 按功能导航

预期提升：

- 直接定位评论、状态流转、统计、activity 等功能链路
- 减少无关文件读取
- 更清楚地映射前端、API、后端 route、storage、tests

主要对比：

- R22/R23 的文件读取数
- 链路解释准确度
- 是否遗漏关键环节或误判不确定点

### 3. 范围控制

预期提升：

- 减少 R14/R15 这类“识别目标已完成但仍修改旁支”的情况
- 小维护任务 diff 更小

主要对比：

- `scopeDriftCount`
- `filesChanged`
- `overImplementedFutureTasks`

### 4. 修复和验证纪律

预期提升：

- 更少的后期阻断问题
- 更早发现 build/runtime 风险
- 更清楚地区分功能完成与验证完成

主要对比：

- `commandFailureCount`
- repair rounds
- human verification failures

### 5. 开发者认知

预期提升：

- 降低黑盒感
- 维护入口更清楚
- 修改某个功能时更有信心

这部分带有主观性，但应作为补充评估维度记录。

## 推荐的 CoAI 测试设置

使用同一个 TeamDesk Lite 任务序列运行 CoAI 条件组。

主开发序列：

- 使用同样的 R01-R20 prompt
- 适当加上 `$coai-skill`
- 记录同样的 `runs.csv` 指标
- 不修改产品 README/spec

新窗口序列：

- 在项目已存在后开启新 chat
- 运行 R21-R25
- CoAI 条件允许读取 `.coai/project`、`.coai/mapper`、`.coai/node`
- 与 baseline R21-R25 对比

最重要的 CoAI 证据会来自 R21-R25，而不是证明项目能不能被开发出来。

## 当前 Baseline 判定

baseline 判定：

> 足以完成产品，但暴露出认知成本、早期返工成本、范围漂移和功能地图黑盒问题。

CoAI 机会：

> 改善项目认知、维护导航和新窗口上下文连续性，而不是证明基础任务可完成。
