# CoAI Core（中文版）

## 作用

这个文件只用于“判断与思考”，不用于运行时操作。

英文 `SKILL.md` 是给 agent 使用的主版。

`SKILL_ch.md` 是给中文开发者审校、补充和优化措辞的镜像版。

它的职责是判断：

- 某个改动归属于哪个 module / feature
- 某个能力是旧功能的一部分，还是新功能
- `.coai` 是否需要新增或更新
- 共享代码应继续保留业务功能认知，还是升格为 `utils/common`

## 高密度 Core 规则

在做任何边界判断前，先记住这些规则：

- 先判 feature，再生成资产，再进入实现
- 先判 feature，再判源码文件归属，不能反过来
- feature 按一个连续场景定义，不按文件名、helper 名或某段实现细节定义
- 源码文件按实现职责隔离定义，不按功能文档长相定义
- 共享代码不会因为目录位于 `utils/common` 就自动升格为 `utils/common`
- 只有当共享能力本身成为开发者需要直接理解的认知入口时，才升格 common
- 只有认知层变化时才更新 `.coai`，不是每次内部重构都要动
- 在写代码前，应先稳定 `module / feature -> source file -> node semantics -> anchor id -> mapper skeleton`

## 快速 Core 导航

如果当前只差一个判断，用这个导航：

- 判断这是旧 feature 还是新 feature：走功能边界判断链
- 判断代码留在原文件还是拆新文件：走源码文件归属判断链
- 判断共享代码保留业务认知还是升格 `utils/common`：走共享代码升格判断链
- 判断 `.coai` 到底要不要动：走 `.coai` 变更清单

## 功能边界判断链

最短判断顺序：

1. 先识别这次改动服务的是哪个场景
2. 检查目标、触发时机、处理链路是否仍与某个已有 feature 一致
3. 如果一致，继续归入原 feature
4. 只有三者明显分离时，才拆成新 feature

## 源码文件归属判断链

最短判断顺序：

1. 先确认 feature 归属
2. 再看实现是否还能留在当前文件而不持续扰动稳定模块
3. 如果可以，就留在当前文件
4. 如果不可以，就按实现职责隔离拆新文件

## 共享代码升格判断链

最短判断顺序：

1. 先判断这段代码是不是只是被多个业务功能复用的共享实现
2. 如果是，就保留业务功能认知，并在需要时允许多语义 anchor
3. 再判断这个共享能力本身是否已经成为开发者必须直接理解的对象
4. 只有这时，才新增或升格 `utils/common` 认知入口

## 功能边界规则

优先使用这些规则：

- 功能边界按“场景事件的逻辑连续性”定义
- 一个功能可以跨多个源码文件
- 只有当目标、触发时机和处理链路明显分离时，才拆成新功能

当前仓库的主要功能族有：

- `extension-entry`
- `navigation`
- `git-sync`
- `bug-repair`
- `cli`

## 源码文件规则

先判断功能归属，再判断源码文件归属。

- 源码文件边界按“实现职责隔离”定义
- 如果继续留在原文件会反复扰动稳定模块，就应拆新文件
- 当结构已经明确时，应在正式实现前先确定源码文件归属

功能边界和源码文件边界有关联，但不是同一件事。

## 实现前规划规则

在真正写代码前，按这个顺序先稳定下来：

1. module / feature 归属
2. 源码文件归属
3. 节点语义
4. `anchor id`
5. mapper 骨架归属

这条规则的目的，是避免 `.coai` 资产变成“代码写完后的事后追认”。

## module / feature 分类

### module

module 应按稳定业务域或技术域选择。

例如：

- `auth`
- `order`
- `plugin`
- `common`

不要只因为文件路径在某个目录下，就机械地把 module 也定成那个目录名。

### feature

feature 必须对应一个连续场景，而不是：

- 某个文件名
- 某个技术点
- 某个孤立 helper 函数

好的例子：

- `register`
- `checkout`
- `git-sync`
- `bug-repair`

不好的例子：

- `utils`
- `controller`
- `api-helper`

除非“共享能力本身”已经成为明确认知对象。

## 共享代码与多语义 anchor 规则

这是强制判断点。

当多个业务节点命中同一段实现时：

- 默认允许在同一代码段附近维护多条不同语义 anchor
- 不要因为代码在共享目录下，就自动升格为 `utils/common` 功能
- 只有当共享能力本身已经成为开发者需要单独理解的认知入口时，才升格为 `utils/common`

### 触发时机

在这些动作之前必须先应用这条规则：

- 给共享代码新增 mapper 条目
- 给共享代码新增或修改 anchor
- 新建 `utils/common` 功能文档
- 把多个业务节点合并成一个 common 认知入口

### 默认决策

如果共享代码只是实现复用：

- 保留业务功能节点
- 允许多条语义 anchor 靠近同一段实现

如果共享代码本身需要被直接理解：

- 新增独立的 `utils/common` 功能文档
- 原业务节点如果仍然有意义，就继续保留
- common 功能是新增一层认知，不是默认替代业务节点

## `.coai` 变更判断

用这个清单判断 `.coai` 是否需要变动。

需要更新 `.coai` 的情况：

- 功能意图变了
- module / feature 归属变了
- anchor 语义变了
- mapper 指向变了
- node 认知变了
- 新增了功能
- `current.md` 应该反映有意义的项目进展

不需要更新 `.coai` 的情况：

- 只是内部重构，但功能认知没变
- 文件移动了，但功能语义没变，系统维护可以恢复映射
- 某个共享 helper 变化了，但功能层理解没变

## `.coai` 职责规则

职责边界要保持清楚：

- AI 负责新增 CoAI 资产
- 系统负责确定性的 mapper 更新和 bug 检测
- AI 或人工负责语义修复判断

层级职责：

- `project`：功能认知与规格
- `node`：Hover 认知
- `mapper`：代码定位
- `current.md`：只记录动态进展
- `log/bugs`：bug 生命周期上下文

## 开发者项目角色边界

当你在思考一个接入 CoAI 的宿主项目时，保持这个模型：

- 认知层产出：`.coai/`
- 源码层产出：宿主项目真实业务代码，可是任意语言/框架
- 分发层产出：宿主项目自己的发布产物

CoAI 改变的是认知层。

它不应该改写宿主项目的语言/框架选择，也不应该变成宿主项目的运行时或最终分发产物。

## references 边界

使用规则：

- `../references/*` 是英文主版参考树
- `../references/ch/*` 是中文镜像参考树

如果仓库里已经存在稳定规则，不要临时凭经验发明新的边界规则。

## 约束

- 不要按源码文件数量划分 feature
- 不要因为共享目录存在，就自动升格 common 认知
- 在功能归属不清楚前，不要先生成 `.coai`
- 不要把认知层变更和运行时变更混为一谈
